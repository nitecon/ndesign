The backend is always authoritative

Thin clients. Dumb templates. Server-owned truth. A year into rebuilding our admin surfaces on top of a hypermedia-first stack, here is what actually changed — and what broke on the way there.

WH

Will Hattingh

Staff engineer · Platform

Why we stopped shipping state to the client

For years our admin surfaces shipped a client-side state machine that mirrored half the domain model. Every new feature started with a question nobody wanted to answer: where does this piece of data live — on the server, in the store, or in the component? The answer was usually all three, and the bugs followed.

The shift was narrative, not technical. Once we agreed that the backend would be the only source of truth, the client’s job collapsed to three things: render markup the server sends, bind inputs to endpoints, and surface feedback. No reducers. No derived state. No cache invalidation dance. Just HTTP and a template.

Heuristic: if the frontend is answering a question the backend could answer, you have duplicated state and it will drift.

What the client is still responsible for

A thin client is not an empty one. It still owns presentation and ergonomics — just nothing that requires knowing the domain. In our stack that distils to:

  • Binding DOM elements to endpoints with small, declarative attributes.
  • Rendering <template> fragments from JSON responses.
  • Surfacing loading, empty, and error states the backend already labeled.
  • Keyboard, focus, and motion — the things a server literally cannot do.

Everything else is a RPC round-trip. The tradeoff is obvious and we made it deliberately: a little latency, a lot less complexity.

The migration, in three passes

  1. Inventory. Audit every client-side store, context, and module-level cache. Most of them exist to dodge a request the team was afraid to make.
  2. Invert. Replace each store with a binding to the endpoint that already answers its question. Delete the store.
  3. Harden. Cache at the edge, not in the client. Stream where it helps. Add feedback affordances so the user never wonders whether a click registered.
“The backend is always authoritative. The frontend is a thin presentation and binding layer delivered via CDN.”
— ndesign Architecture Spec

What a binding looks like

The binding pattern is almost embarrassingly small. A card that renders whatever /api/users/2 returns is one element and one template: 

<div class="nd-card"
     data-nd-bind="${api}/api/users/2"
     data-nd-template="user-tpl">
  <template id="user-tpl">
    <div class="nd-card-header"><h3>{{name}}</h3></div>
    <div class="nd-card-body"><p>{{email}}</p></div>
  </template>
</div>

No JavaScript. No framework. No build step. The same file runs in production, in a static preview, and in the offline docs you are reading right now. That’s the point.

What we measured afterward

We tracked four numbers across the six months before and after the cutover. The client-side numbers are the ones that surprised us — not because they got better, but because they got simpler.

Metric Before After Delta
JS shipped (gzipped) 312 KB 41 KB −87%
Time-to-interactive (p75) 2.8 s 0.9 s −68%
State-sync bugs / quarter 23 2 −91%
Median PR review time 18 h 19 h flat

The flat review time is the honest one: moving complexity to the server does not make the codebase smaller overall, it makes it legible. Our backend PRs got longer; our frontend PRs got shorter; the overall throughput was unchanged. What moved was our ability to reason about a failure after it shipped.

Things we still got wrong

Optimistic updates

For a month we tried to keep optimistic updates on the client. They are the one thing users notice when you remove them. We ended up moving the optimism server-side too — 202 Accepted plus a pending badge — and eating a slightly worse perceived latency for a much smaller surface area.

Keyboard shortcuts

Power users expected Cmd + K command palettes and G D style chord navigation. Those belong fully on the client. We should have built them first instead of last.

Questions we get asked a lot

Isn’t this just server-side rendering?

Not exactly. SSR ships the initial HTML from the server, then hydrates a client-side app that owns state. We never hydrate. Subsequent interactions hit the same endpoints that rendered the page, and we swap fragments back in. Closer to hypermedia than SSR.

What about offline?

We accept that offline is a different product. For admin tools, nobody edits a user table on a plane. If you need offline-first, you need a sync engine, and that is a larger architectural commitment than a CSS framework can meaningfully help with.

Does this scale to public-facing apps?

It scales to any app where latency is tolerable and features are mostly CRUD. The further you get from CRUD — games, editors, real-time collaboration — the more you need a thick client. Pick the right tool per surface; don’t pick one religion for the whole org.

The part worth stealing

You do not need our framework to get most of this. The thing worth stealing is the rule, not the code: one source of truth, and it lives on the server. Every decision downstream of that rule gets easier. Every decision upstream of it is a cache invalidation problem in disguise.

If you try it, start with a single surface. A settings page is a good candidate — low traffic, high form-density, forgiving UX expectations. Inventory the state, invert the bindings, delete the store. Measure the diff. Decide from there.
